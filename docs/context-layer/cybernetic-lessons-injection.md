# HRCN v0.1.2 — Cyber/CMS/RCC-N Lesson Injection

Status: docs/context/navigation layer only.

This file injects the durable lessons learned from CMS-SA v0.4.5 through v0.5.1 and RCC-N v1.7 into the Hermes HRCN context layer.

## Purpose

Hermes already provides the actor/runtime body. This lesson injection records the governance lessons required before Hermes is connected to CMS-style memory, repair, dry-run, apply-gate, rollback, and permission surfaces.

## RCC-N v1.7 Adoption Injection

```text
RCC tells the agent what the repository means.
RCC-N tells the agent where it is.
Validation tells the agent whether reality agreed.
Profiled geometry is adoptable governance.
```

HRCN v0.1.2 adopts a Lite-to-Standard documentation profile. It must not over-govern Hermes runtime, but it must prevent under-governance before memory, repair, dry-run, or apply authority is attempted.

## Injected Lessons

### HRCN-L-004 — Evidence-level lessons must become public surface memory.

- Source: CMS-SA v0.5.1
- Hermes/HRCN application: Important CMS and RCC lessons must be logged in Hermes HRCN docs before runtime integration.

### HRCN-L-005 — Final verifier residue must be committed or discarded.

- Source: CMS-SA v0.5.1 residual closure
- Hermes/HRCN application: Hermes HRCN push scripts must not pretend the repo is clean when generated evidence or local artifacts remain.

### HRCN-L-006 — Agent proposal is not authority.

- Source: CMS-SA v0.5.0
- Hermes/HRCN application: Hermes may propose actions; CMS/HRCN decides permission class; human authorization gates dangerous writes.

### HRCN-L-007 — Read-only context is not runtime integration.

- Source: CMS-SA v0.5.0
- Hermes/HRCN application: docs/context-layer can orient Hermes without changing prompts, memory, tools, providers, or runtime behavior.

### HRCN-L-008 — Dry-run is not write authority.

- Source: CMS-SA v0.4.5
- Hermes/HRCN application: Future Hermes repair plans must stay simulation-only until explicit apply gates and authorization exist.

### HRCN-L-009 — Apply gate, apply packet, and dry-apply sandbox are not live authority.

- Source: CMS-SA v0.4.6-v0.4.8
- Hermes/HRCN application: Each future HRCN bridge phase must preserve blocked actions and rollback evidence before any live target write.

### HRCN-L-010 — Validation order is part of system geometry.

- Source: CMS-SA v0.4.5
- Hermes/HRCN application: HRCN must validate phase boundaries in order: context -> route map -> proposal -> dry-run -> evidence -> authorization.

### HRCN-L-011 — Registry/history must not lag current checkpoint.

- Source: CMS-SA v0.5.1
- Hermes/HRCN application: Hermes HRCN roadmap and validation reports must advance together.

### HRCN-L-012 — File-run scripts prevent interactive paste drift.

- Source: CMS failure ledger
- Hermes/HRCN application: Large Hermes HRCN patches should be delivered as file-run PowerShell scripts.

### HRCN-L-013 — Profile adoption is not validation.

- Source: RCC-N v1.7
- Hermes/HRCN application: HRCN should select the lightest sufficient governance profile and measure utility before claiming improvement.

## Non-Claim Lock

It does not modify Hermes runtime behavior, prompts, tools, skills, plugins, providers, gateways, TUI, web runtime, package dependencies, memory authority, security posture, production readiness, autonomy, consciousness, sentience, AGI/ASI status, or external validation.
